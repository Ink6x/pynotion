# ロードマップ — Phase 1〜4

工数は「ポートフォリオ品質(テスト込み)」の目安。1 人日 = 実働 6〜7h。
各 Phase は独立してマージ・デモ可能になるよう構成する。

## Phase 1 — 「実用アプリ」化(5,500 円帯の土台) / 約 11〜15 人日

| 項目 | 依存 | リスク | 工数 |
|---|---|---|---|
| A. 認証・マルチユーザー・RBAC(owner + 基本共有) | なし | 既存 46 テストの全面改修(中) | 5〜7 人日 |
| I. Docker 化・settings 分割・本番安全化・GitHub Actions CI | なし | 低 | 3〜4 人日 |

**狙い**: 「他人に使わせられる・デプロイできる・CI が回る」状態。
5,500 円帯の最低ラインをクリア。

**リスク対策**: A は全 API テストに影響するため、
**最初に `authenticated_client` フィクスチャを整備**してから着手する。

## Phase 2 — 「設計力・スケール基礎」(6,000 円帯を確実に) / 約 13〜18 人日

| 項目 | 依存 | リスク | 工数 |
|---|---|---|---|
| B. PostgreSQL 移行 + pg_trgm / SearchVector 全文検索 | I(settings 分割) | データ移行(中) | 3〜4 人日 |
| H. API 体系化 + OpenAPI(django-ninja)+ レート制限 | A | 既存互換(中) | 3〜5 人日 |
| K. Playwright E2E | Phase 1 完了 | 低 | 3〜4 人日 |
| J. N+1 計測・キャッシュ・debug-toolbar | B | 低 | 2〜3 人日 |

**狙い**: 適切な DB・全文検索・文書化 API・E2E・計測。6,000 円帯を確実に。

## Phase 3 — 「難問を解いた」証明(8,000 円帯への跳躍) / 約 17〜25 人日 — ✅ 完了

| 項目 | 依存 | リスク | 工数 |
|---|---|---|---|
| E. ブロックのネスト(再帰構造・Tab/Shift+Tab) | なし | エディタ改修(中) | 4〜6 人日 |
| C-3a. リアルタイム同期(Channels + 楽観ロック + ブロードキャスト) | A, I(Redis) | WebSocket 認証・再接続(高) | 5〜7 人日 |
| F. バージョン履歴 | E | 低 | 4〜5 人日 |

**狙い**: 並行性・再帰ドメイン・履歴という難所の実装。
8,000 円帯の説得力が出る最初の段階。

## Phase 4 — 「最高難度」と運用の完成(8,000 円帯を盤石に) / 約 20〜30 人日

| 項目 | 依存 | リスク | 工数 |
|---|---|---|---|
| C-3b. 文字単位の競合解決(Yjs / y-py CRDT) | C-3a | 最高(CRDT 統合・永続化) | 7〜10 人日 |
| D. データベースビュー(スキーマレス・table/board) | A, B, E | 高(動的クエリ) | 8〜12 人日 |
| G. 非同期エクスポート / Webhook(RQ) | I(Redis) | 中 | 5〜7 人日 |
| 運用仕上げ(Sentry・構造化ログ・負荷試験数値) | I | 低 | 2〜3 人日 |

**狙い**: 協調編集の真の難所と Notion の核心機能。8,000 円帯超も視野。

## 依存関係グラフ

```
Phase 1:  A (認証/RBAC) ──┬──→ H (OpenAPI)        [Phase 2]
          I (Docker/CI) ──┼──→ B (PostgreSQL) ──→ J (計測)   [Phase 2]
                          │         │
                          │         └──→ D (DB ビュー)        [Phase 4]
                          ├──→ C-3a (リアルタイム) ──→ C-3b (CRDT) [Phase 3→4]
                          └──→ G (非同期/Webhook)             [Phase 4]
          E (ネスト) ──→ F (履歴)、D                       [Phase 3]
          K (E2E) は Phase 1 完了後いつでも                  [Phase 2]
```

## 全体リスク

| リスク | レベル | 対策 |
|---|---|---|
| 認証導入による既存テストの大規模改修 | 中 | フィクスチャ先行整備、API の互換維持 |
| WebSocket の認証・再接続・状態同期 | 高 | C-3a(ブロック粒度)で先に基盤を固め、C-3b は別 Phase に分離 |
| CRDT(Yjs/y-py)と Django の統合 | 最高 | スパイク(検証実装)を先に行い、ダメなら C-3a 止まりでも成果になる |
| スキーマレス動的クエリの安全性 | 高 | ORM の `Q` 経由のみ、演算子をホワイトリスト化 |
| スコープ肥大化 | 中 | 各 Phase 完了ごとにデモ可能・README 更新を必須にする |
