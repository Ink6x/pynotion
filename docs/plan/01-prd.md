# PRD — pynotion ポートフォリオ強化

## 1. 目的

Django 製 Notion クローン「pynotion」を、フリーランス市場で
**時給 5,500〜6,000 円帯、最終的に 8,000 円帯** のエンジニアであることを
証明するポートフォリオへ育てる。

その価格帯に期待されるスキル(設計力・スケーラビリティ・運用視点・
セキュリティ・複雑なドメインロジック)から逆算し、機能と設計を追加する。

## 2. 現状評価

### 既に強みになっている設計

| 項目 | 内容 |
|---|---|
| fractional indexing | `pages/ordering.py` — midpoint 方式の自前移植。並べ替え時に他行を更新しない。協調編集を見据えた並び順設計として語れる |
| ソフトデリート設計 | 削除時刻マーカーによる復元単位の保持、孤児ページのルート付け替え(`pages/models.py`) |
| API 封筒 | `{ok, data, error}` 統一 + エラーの一元変換(`pages/http.py`) |
| インデックス設計 | `(parent, is_deleted, position)` / `(page, position)` の複合インデックス |
| テスト | pytest 46 件、カバレッジ 96% |

### 頭打ちの要因

| 項目 | 問題 |
|---|---|
| 認証 | ゼロ。シングルユーザーで全 API が無防備 |
| DB | SQLite 固定。検索が `icontains`(日本語で実用性が低い) |
| リアルタイム | 同期なし。ASGI/Channels 未導入 |
| ブロック構造 | フラット。ネスト(toggle 等)不可 |
| 運用 | CI/CD・Docker なし。`DEBUG` デフォルト 1、SECRET_KEY フォールバックあり |
| E2E | なし。自作 contenteditable エディタはユニットテストで担保しづらい |

## 3. 単価帯別ギャップ分析

### 時給 5,500〜6,000 円帯に期待されるシグナル

| 期待シグナル | 現状 | ギャップ |
|---|---|---|
| 認証・認可を含む実用 Web アプリを一人で完結 | 認証なし | **大** |
| RDB の適切なモデリング・インデックス・マイグレーション運用 | 良好だが SQLite 固定 | 中 |
| REST API の一貫した設計とエラーハンドリング | 封筒設計は良好 | 小(文書化が必要) |
| テストを書く習慣 | 96% | 小(E2E が欠落) |
| Docker・基本的な CI | なし | 中 |

→ 設計の素地は十分。**認証なし・SQLite・CI/Docker なしが決定的に足を引っ張る**。

### 時給 8,000 円帯に期待されるシグナル

| 期待シグナル | 現状 | ギャップ |
|---|---|---|
| 並行性・競合解決など「難問」を設計できる | 未着手(fractional indexing の素地あり) | **大** |
| スケーラビリティ(全文検索・N+1・キャッシュ・非同期) | 未着手 | 大 |
| 複雑なドメインモデル(スキーマレス・再帰構造) | フラット構造 | 大 |
| 運用視点(監視・ログ・Sentry・ヘルスチェック) | なし | 大 |
| RBAC・マルチテナント・共有モデル | なし | 大 |

→ 「リアルタイム共同編集の競合解決」「PostgreSQL 全文検索」
「データベースビュー(スキーマレス)」「運用基盤」という難所の実績が効く。
**fractional indexing を既に持っていることが最大のアドバンテージ**。

## 4. 成功基準

### Phase 1〜2 完了時(5,500〜6,000 円帯) — ✅ 達成

- [x] 認証 + ページ共有/権限(RBAC)があり、他人に使わせられる
- [x] PostgreSQL で動作し、日本語全文検索が実用レベル(pg_trgm + SearchVector)

- [x] OpenAPI ドキュメントが自動生成される(django-ninja、`/api/docs`)

- [ ] ブロックの無限ネスト(toggle / Tab インデント)
- [ ] バージョン履歴と復元
- [ ] データベースビュー(table/board、ユーザー定義プロパティ)

## 5. 面談で語る技術ストーリー(要点)

- **最大の差別化トーク**: 「協調編集の競合解決を、ブロック粒度の楽観ロックから
  文字粒度の CRDT へ段階的に設計。既存の fractional indexing がシーケンス CRDT の
  位置識別子と同系統であることを活かし、並べ替えの衝突を構造的に回避した」
- 「LIKE 検索の限界を踏まえ、日本語に形態素解析なしで対応するため
  pg_trgm + SearchVector のハイブリッドを選定した」
- 「DRF 全面移行ではなく既存の軽量封筒設計を尊重し、django-ninja で
  OpenAPI を自動生成。"作り直さない"技術選定をした」
- 「contenteditable ベースの自作エディタはユニットで担保しづらいため、
  Playwright で主要フローを E2E 化し CI に組み込んだ」
